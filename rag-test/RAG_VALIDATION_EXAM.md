### A1
互斥锁的保护范围中，哪些操作原则上不得放入？

**答案：**

根据规范 §2.1.1「尽量缩小临界区」，互斥锁的临界区内原则上不得放入以下操作：

1. **I/O 操作**：磁盘读写、文件操作
2. **网络操作**：HTTP 请求、Socket 通信
3. **复杂计算**：CPU 密集型算法处理
4. **外部对象方法调用**：调用外部回调或其他对象的方法（§2.1.2）
5. **日志写入**：大量或同步的日志输出
6. **发布消息**：如 ROS2 `publisher_->publish()` 调用

核心原则是：锁应仅保护共享状态的访问，而非整个业务流程。在临界区内执行耗时操作会显著增加锁竞争，导致高优先级任务被阻塞，特别是在 ROS2 实时系统中会引发回调堆积问题。

### A2
为什么多把锁时必须建立全局锁顺序？违反后会怎样？

**答案：**

根据规范 §2.1.3「锁顺序必须全局一致」：

**为什么必须建立全局锁顺序：** 当一个模块需要同时持有多把锁时，如果不同线程以不同顺序获取这些锁，就会产生**死锁（Deadlock）**的风险。全局锁顺序确保所有线程以相同的顺序获取锁，从而从根本上消除循环等待条件。

**违反后的后果：**
1. **死锁**：线程 A 持有锁1等待锁2，线程 B 持有锁2等待锁1，两者永久阻塞
2. **系统挂起**：死锁发生后相关线程无法恢复，可能导致整个系统不可用
3. **难以排查**：死锁通常难以在测试中复现，往往在生产环境中偶发

**最佳实践：** 为所有锁建立全局编号或层级，所有线程必须按编号从小到大的顺序获取锁。优先使用 `std::scoped_lock` 同时获取多把锁以避免手动排序。

### A3
为什么条件变量必须使用谓词等待？

**答案：**

根据规范 §2.3.1「必须使用谓词等待」：

条件变量必须使用谓词等待（而非裸 `cv_.wait(lock)`），原因如下：

1. **虚假唤醒（Spurious Wakeup）**：操作系统和线程库的实现允许线程在没有 `notify` 的情况下被唤醒。裸等待无法检测这种情况，会导致线程在条件未满足时继续执行。

2. **信号丢失（Lost Wake-up）**：如果 `notify` 在 `wait` 之前发生，且没有谓词检查前置条件，线程可能永久等待一个已经发生的事件。

3. **正确写法应为：**
   ```cpp
   cv_.wait(lock, [this]{ return condition_is_met_; });
   ```
   谓词等待模式会在每次唤醒时自动检查条件，只有条件为 true 时才继续执行，确保代码的正确性和可维护性。

### A4
ROS2 订阅回调中禁止做哪些长阻塞操作？

**答案：**

根据规范 §3（ROS2 编码规范）及 §7（实时性与性能规范），ROS2 订阅回调中禁止做以下长阻塞操作：

1. **磁盘 I/O**：文件读写、日志落盘（`file_writer_->Write()`）
2. **网络操作**：HTTP 请求、Socket 发送/接收
3. **线程睡眠**：`std::this_thread::sleep_for()` 或任何显式等待
4. **重计算**：CPU 密集型算法处理（如图像处理、路径规划）
5. **同步阻塞调用**：等待其他线程完成、同步服务调用
6. **长时间持锁操作**：在锁保护范围内执行耗时逻辑

**原因：** ROS2 回调在 Executor 线程中执行，阻塞回调会导致回调堆积，影响其他订阅和 Timer 的执行时效。高频回调（如 30Hz/100Hz）中的阻塞尤其危险，会造成消息丢失和系统响应延迟。

**正确做法：** 回调只做轻量操作（数据拷贝、状态更新），将耗时任务卸载到独立工作线程。

### A5
参数动态更新时为什么要用快照语义？

**答案：**

根据规范 §4.1.2「参数更新应使用快照语义」：

参数动态更新时必须使用快照语义，原因如下：

1. **避免"半更新配置"问题**：如果工作线程在处理过程中分别读取多个参数（如 timeout_ms、max_retries、enable_feature），而参数在两次读取之间被更新，工作线程会读到新旧混合的配置状态，导致不可预测的行为。

2. **保证一致性**：快照语义要求将所有相关参数组织为一个结构体（Config struct），在更新时生成完整快照，工作线程仅读取快照中的一致性配置，避免竞态条件。

3. **实现方式：**
   - 将相关参数打包为 `struct Config`
   - 使用 `std::shared_ptr<const Config>` 持有配置
   - 更新时原子替换整个 `shared_ptr`
   - 工作线程拷贝 `shared_ptr` 后使用，保证整个处理周期内配置一致

### A6
为什么不要在业务类内直接new外部依赖？

**答案：**

根据规范 §6.1.1「不要在业务类内部直接 new 外部依赖」：

不应在业务类内部直接实例化外部依赖（如 Clock、Logger、FileWriter、MetricsClient 等），原因如下：

1. **可测试性差**：直接 new 的外部依赖无法在单元测试中被替换为 mock 对象，导致测试必须依赖真实的文件系统、网络、时钟等外部系统，测试成本极高。

2. **耦合度高**：业务逻辑与具体实现绑定，修改外部依赖时需要同步修改业务类。

3. **不可控**: 外部依赖的行为（如网络超时、文件不存在）在测试中不可控，导致测试结果不确定。

**正确做法：** 通过构造函数注入接口或轻量包装对象（依赖注入 DI 模式），使业务类仅依赖抽象接口，测试时传入 mock 实现。

### A7
错误日志必须包含哪些信息才能有效定位问题？

**答案：**

错误日志必须包含以下关键定位信息：

1. **输入参数**：触发错误时的关键输入值（如传入的 frame_id、message 类型）

5. **错误码/错误类型**：具体的错误分类，而非泛化描述
6. **时间戳**：精确到毫秒的错误发生时间

### A8
无上限队列缓存的风险是什么？

**答案：**

根据规范 §7.2「背压与队列管理」：

无上限队列缓存的主要风险包括：

1. **内存耗尽（OOM）**：当生产速度持续大于消费速度时，队列无限增长，最终耗尽系统内存，导致进程被 OS 杀死或系统崩溃。

2. **延迟不可控**：队列中积压的旧消息使消费者处理的数据严重过时，在实时系统中意味着使用过期数据做决策（如自动驾驶中使用几秒前的传感器数据）。

3. **级联故障**：内存压力触发频繁 GC 或 swap，导致整个系统性能急剧下降，可能引发连锁反应。

4. **问题难以发现**：在低负载测试中表现正常，只有在生产环境中高负载时才暴露，难以预测和复现。

**应对措施：** 设置队列容量上限、定义溢出处理策略（丢弃旧消息/拒绝新消息）、实现背压机制、统计丢包与性能指标并发布诊断告警。

### A9
诊断状态与业务状态为什么要分离？

**答案：**

根据规范 §5.2.1「诊断状态与业务状态分离」：

诊断状态（internal diagnostic status）与业务状态（external business object status）必须分离，原因如下：

1. **职责不同**：业务状态描述"业务对象当前处于什么阶段"（如订单已支付），诊断状态描述"系统运行是否正常"（如传感器连接状态、处理延迟）。混合两者会导致语义模糊。

2. **变更频率不同**：诊断状态可能频繁变化（每秒更新），而业务状态变更由外部事件驱动。混合后难以独立监控。

3. **消费者不同**：业务状态面向业务决策和用户；诊断状态面向运维、监控系统和自动告警。将诊断塞入业务状态会污染业务数据。

4. **可观测性**：独立的诊断通道方便接入标准的监控告警体系，实现自动化运维。

**最佳实践：** 使用独立的诊断 Publisher（如 ROS2 DiagnosticArray），定义标准的诊断等级（OK / WARN / ERROR / STALE），包含可解释的上下文信息（§5.2.2）。

### B1
在互斥锁保护状态下，调用外部对象的方法有什么风险？

**答案：**

根据规范 §2.1.2「禁止拿锁后调用外部代码」及 §2.1.1「尽量缩小临界区」：

在互斥锁保护状态下调用外部对象的方法存在以下重大风险：

1. **死锁风险**：外部方法可能尝试获取其他锁，甚至反向获取当前锁，形成循环等待导致死锁。调用者无法控制外部代码内部的锁行为。

2. **回调反转（Callback Inversion）**：外部方法可能再次调用持锁模块的其他接口，导致同一把锁被同一线程重入（若非递归锁则直接死锁）。

3. **临界区膨胀**：外部方法的执行时间不可预测（可能包含 I/O、网络、复杂计算），导致锁持有时间过长，增加锁竞争，阻塞其他线程。

4. **行为不可推理**：外部代码的实现可能随版本变化，引入新的阻塞或锁操作，使系统行为变得不确定。

**正确做法：** 先在锁内完成数据拷贝/状态更新，释放锁后再调用外部方法。

### B2
高频ROS2回调中为什么要减少动态内存分配？

**答案：**

根据规范 §3.2.2「回调里不要做不必要的动态内存分配」及 §7.1「高频路径要求」：

高频 ROS2 回调中必须减少动态内存分配，原因如下：

1. **分配器延迟不确定**：`new`/`malloc` 的执行时间不确定，可能触发操作系统的页面分配、内存碎片整理，导致回调延迟波动，破坏实时性。

2. **锁竞争**：标准分配器（如 glibc malloc）内部有全局锁，高频调用导致多线程竞争分配器锁，降低吞吐量。

3. **缓存不友好**：频繁分配和释放小对象导致内存碎片化，降低 CPU 缓存命中率，影响处理性能。

4. **GC 压力**：频繁的 new/delete 增加内存管理开销，在 30Hz/100Hz 回调中这些开销会被放大数十倍。

**正确做法：** 使用预分配缓冲区、对象池复用、`reserve()` 预留容器容量，确保高频路径零分配。

### B5
处理函数为什么要优先显式返回结果，而不是修改成员变量？

**答案：**

根据规范 §6.2「可测试函数设计」及 §6.2.1「纯逻辑与 ROS 接口层分离」：

处理函数应优先显式返回结果而非修改成员变量，原因如下：

1. **可测试性**：显式返回值的函数可以通过 `EXPECT_EQ(result, expected)` 直接验证，无需额外检查对象内部状态。修改成员变量则需要通过 getter 方法间接验证，增加测试复杂度。

3. **并发安全**：显式返回值不涉及共享状态修改，天然线程安全；而修改成员变量需要额外的锁保护。

4. **可组合性**：返回值可以直接作为其他函数的输入，形成函数管道；副作用式函数则必须按特定顺序调用。

5. **独立性**：函数不依赖对象的状态历史，可以独立调用和测试，降低耦合度。

### B6
为什么高频路径中的日志必须节流？

**答案：**

根据规范 §7.1「高频路径要求」及 §3.2「回调执行规范」：

高频路径中的日志必须节流，原因如下：

1. **I/O 阻塞**：日志写入涉及磁盘 I/O 或终端输出，是阻塞操作。在 100Hz 回调中每次都写日志会严重拖慢回调执行速度。

2. **锁竞争**：日志框架内部通常有互斥锁保护日志缓冲区，高频写入导致多线程争抢日志锁，引发回调延迟。

3. **磁盘带宽饱和**：100Hz × 每条日志几百字节 = 每秒几十 KB 日志数据，长时间运行会快速消耗磁盘空间和写入带宽。

4. **信息淹没**：过量日志使真正重要的告警和错误信息被淹没，降低日志的诊断价值。

**正确做法：**
- 使用 `RCLCPP_INFO_THROTTLE` 等节流宏，限制日志频率（如每秒最多 1 次）
- 高频路径仅记录 WARN/ERROR 级别日志
- 统计类信息通过诊断发布而非日志输出

### B7
发现异常或失败返回时应该如何处理？

**答案：**

根据规范 §5.1「错误处理」及 §5.1.2「错误日志要携带可定位信息」：

发现异常或失败返回时，应按以下步骤处理：

1. **明确捕获，禁止吞掉异常**：不得使用空 `catch(...)` 或忽略返回值。每个错误分支都必须有明确的处理逻辑。

3. **更新诊断状态**：将模块诊断状态从 OK 切换为 WARN 或 ERROR，并附带可解释的上下文信息（哪个组件、什么原因、影响范围）。

4. **选择错误策略**：
   - **降级运行**：使用上一次的有效结果或默认安全值
   - **重试**：对瞬态错误进行有限次重试
   - **传播错误**：将错误返回给调用方处理
   - **安全停止**：关键路径上的不可恢复错误应触发安全停机

### B8

### B9
如何区分一个topic需要"最新值"还是"全量事件"语义？

**答案：**

根据规范 §3.3.2「明确"最新值语义"还是"全量事件语义"」：

区分 topic 需要哪种语义，应基于以下判断标准：

**最新值语义（Latest-value）：**
- 消费者只关心当前最新状态，不关心中间值
- 典型场景：传感器位姿、速度估计、环境温度、电池电量
- 丢失中间帧不影响业务正确性
- QoS 配置：`queue_depth=1`，`BEST_EFFORT`
- 缓存结构：单一变量覆盖写入

**全量事件语义（All-events）：**
- 每一帧数据都必须被处理，不可丢失
- 典型场景：用户命令、事件日志、交易记录、控制指令序列
- 丢失任何一条都会导致逻辑错误或状态不一致
- QoS 配置：`queue_depth` 适当增大，`RELIABLE`
- 缓存结构：FIFO 队列，需要背压机制

**判断原则：** 问自己「如果丢了一帧中间数据，下游结果会错吗？」——如果不会错，用最新值语义；如果会错，用全量事件语义。这个决策直接影响缓存结构、队列长度配置、回调处理逻辑和下游状态机的设计方式。

### C1
一个ROS2节点同时负责传感器订阅、数据融合、轨迹规划、诊断聚合、落盘和Web接口输出。

这个设计存在哪些问题？如何重构？
 
**答案：**

根据规范 §6.2.1「纯逻辑与 ROS 接口层分离」、§3.1.2「不要把所有逻辑写在回调里」及 §9.3「场景理解类」：

**存在的问题：**

1. **单一职责违反（God Node）**：一个节点承担了6种不同职责（传感器订阅、数据融合、轨迹规划、诊断聚合、落盘、Web接口），严重违反单一职责原则，导致代码膨胀、维护困难。
                                                                    
2. **可测试性极差**：所有功能耦合在一个类中，无法对融合算法或规划逻辑单独测试，必须启动完整的 ROS2 节点环境。

3. **性能风险**：落盘（磁盘 I/O）和 Web 接口（网络 I/O）是阻塞操作，与高频传感器回调共享 Executor 线程会导致回调堆积和延迟。

4. **故障传播**：Web 接口崩溃可能影响传感器订阅和轨迹规划，任一模块的异常会波及整个节点。

5. **资源竞争**：多个功能共享同一组锁和内存，增加死锁和竞态条件的概率。

**重构方案：**

1. **按职责拆分节点**：
   - `SensorInterfaceNode`：仅负责传感器订阅和数据收发
   - `FusionProcessorNode`：数据融合处理
   - `PlannerNode`：轨迹规划
   - `DiagnosticAggregatorNode`：诊断聚合与发布
   - `RecorderNode`：数据落盘（独立线程处理 I/O）
   - `WebBridgeNode`：Web 接口输出

2. **纯逻辑与 ROS 接口分离**：融合算法、规划算法抽为纯 C++ 类（Processor），不依赖 ROS2 Node，节点仅做消息收发和生命周期管理。

### C2
一个100Hz订阅回调中：
- 对消息做合法性检查
- 更新mutex保护的内部缓存
- 调用Publisher发送处理结果  
- 记录详细日志

这个设计会带来什么问题？

**答案：**

根据规范 §3.2「回调执行规范」、§7.1「高频路径要求」及 §2.1.1「尽量缩小临界区」：

这个 100Hz 回调设计存在以下问题：

1. **锁内发布消息（违反 §2.1.2）**：如果 `publisher_->publish()` 在 mutex 保护范围内执行，则 publish 的内部序列化和网络发送操作会膨胀临界区，增加锁持有时间，阻塞其他线程对缓存的访问。

2. **高频日志 I/O 阻塞（违反 §7.1）**：100Hz 意味着每秒 100 条详细日志。日志写入涉及磁盘 I/O 或终端输出，是阻塞操作，会导致回调执行时间波动，可能超过 10ms 的时间窗口，引发回调堆积。

3. **日志锁竞争**：日志框架内部通常有全局锁，100Hz 频率下多线程争抢日志锁会进一步增加延迟。

4. **回调过重**：合法性检查 + 加锁缓存更新 + 发布 + 日志，四个操作叠加可能导致单次回调耗时过长，100Hz 下没有足够的时间余量。

### C3
一个模块有3把互斥锁（config_mutex, state_mutex, cache_mutex），在不同函数中被交叉获取。某处代码先拿state后拿config，另处先拿config后拿state。

为什么这很危险？应该怎么修复？

**答案：**

根据规范 §2.1.3「锁顺序必须全局一致」及 §2.1.4「优先用 RAII 管理锁生命周期」：

**为什么危险：**

这是经典的**死锁模式**。当两个线程同时执行不同函数时：
- 线程 A 执行函数1：先获取 `state_mutex` → 再请求 `config_mutex`
- 线程 B 执行函数2：先获取 `config_mutex` → 再请求 `state_mutex`

两个线程形成**循环等待**：A 持有 state 等待 config，B 持有 config 等待 state，永久阻塞。3 把锁的交叉获取使得死锁组合更多（config↔state、config↔cache、state↔cache），问题更难排查。

**修复方案：**

1. **建立全局锁顺序**：为所有锁定义固定编号，如 `config_mutex(1) < state_mutex(2) < cache_mutex(3)`，所有函数必须按从小到大的顺序获取锁。

2. **使用 `std::scoped_lock`**：需要同时持有多把锁时，使用 `std::scoped_lock(config_mutex, state_mutex)` 让标准库自动处理锁顺序，避免手动排序的错误：
   ```cpp
   std::scoped_lock lock(config_mutex_, state_mutex_, cache_mutex_);
   ```

3. **减少同时持有的锁数量**：重新设计数据结构，尽量避免同时需要多把锁的场景。在锁内拷贝数据，释放锁后再操作其他受保护的资源。

4. **代码审查规则**：在 CR 中强制检查多锁获取顺序是否符合全局约定。

### C4
某个模块有多个动态参数（timeout_ms, max_retries, enable_feature）。工作线程在处理时分别读取这三个参数，中间有参数动态更新。

可能出现什么问题？

**答案：**

根据规范 §4.1.2「参数更新应使用快照语义」：

**可能出现的问题——"半更新配置"（Torn Read）：**

工作线程在处理一个请求时分三次读取参数：
1. 读取 `timeout_ms = 5000`（旧值）
2. ← **此时参数动态更新：timeout_ms=1000, max_retries=5, enable_feature=true**
3. 读取 `max_retries = 5`（新值）
4. 读取 `enable_feature = true`（新值）

结果工作线程使用了 `timeout_ms=5000 + max_retries=5 + enable_feature=true` 这种**新旧混合的不一致配置**，可能导致：

- **逻辑矛盾**：旧的长超时与新的高重试次数组合，导致单次请求耗时远超预期（5000ms × 5次 = 25秒）
- **业务错误**：`enable_feature=true` 激活了新功能，但 `timeout_ms` 仍为旧值，新功能在过长的超时下表现异常
- **难以复现**：问题只在参数更新恰好发生在两次读取之间时出现，属于竞态条件

**正确做法：** 使用快照语义：

### C5
一个Producer线程向queue投递数据，Consumer线程处理。Producer速度常常大于Consumer。当前队列无上限增长。

这会导致什么问题？应该设计哪些机制？

**答案：**

根据规范 §7.2「背压与队列管理」及 §3.3.2「明确"最新值语义"还是"全量事件语义"」：

**无上限队列增长导致的问题：**

1. **内存耗尽（OOM）**：Producer 持续快于 Consumer，队列无限增长，最终耗尽系统内存，进程被 OS 杀死（OOM Killer），导致整个节点崩溃。

2. **延迟不可控**：队列中积压大量旧数据，Consumer 处理的数据严重过时。在实时系统中（如自动驾驶），使用几秒前的传感器数据做决策是致命的。

3. **级联雪崩**：内存压力导致系统频繁 swap、GC，整体性能急剧下降，Consumer 处理速度更慢，形成正反馈恶性循环。

4. **故障隐蔽**：低负载测试时不暴露问题，只在生产环境高峰时触发，难以预测和复现。

**应该设计的机制：**

1. **队列容量上限**：设置固定大小的有界队列（如 `boost::lockfree::spsc_queue<T, capacity<1024>>`）。

2. **溢出策略（背压）**：
   - **丢弃最旧数据**（最新值语义）：覆盖队头数据，保证 Consumer 总是处理最新信息
   - **拒绝新数据**（全量事件语义）：Producer 阻塞或返回失败，由上游控制流量
   - **降采样**：每 N 帧只入队 1 帧

3. **丢包统计与告警**：记录丢弃的消息数量，通过诊断发布（DiagnosticArray）暴露队列积压程度和丢包率。

4. **流速匹配**：评估 Consumer 处理能力，合理配置 Producer 频率或增加并行 Consumer。

### C8
某个Service服务处理请求时异常，代码直接catch后返回空结果。

问题在哪？应该怎么做？

**答案：**

根据规范 §5.1「错误处理」及 §5.1.2「错误日志要携带可定位信息」：

**问题分析：**

直接 `catch` 后返回空结果属于**静默失败（Silent Failure）**，是错误处理中最危险的反模式：

1. **错误信息丢失**：异常被吞掉，没有日志记录，运维人员无法知道发生了什么、什么时候发生的、为什么发生。

2. **调用方误判**：调用方收到空结果，无法区分"查询结果确实为空"和"服务处理异常"，可能基于错误前提做出后续决策。

3. **问题难以复现和排查**：没有错误痕迹，当下游出现异常行为时，无法追溯到是这个 Service 的异常导致的。

4. **违反错误处理基本原则**：规范要求每个失败分支都必须有明确的处理逻辑，禁止空 `catch`。

**正确做法：**

```cpp
try {
  auto result = ProcessRequest(request);
  response->set_result(result);
  response->set_success(true);
} catch (const std::exception& e) {
  // 1. 记录可定位的错误日志
  RCLCPP_ERROR(get_logger(),
    "Service [%s] failed: request_id=%s, error=%s, input_param=%s",
    service_name_.c_str(),
    request->request_id.c_str(),
    e.what(),
    request->DebugString().c_str());

  // 2. 更新诊断状态
  UpdateDiagnostic(DiagnosticStatus::ERROR,
    "request processing failed: " + std::string(e.what()));

  // 3. 返回明确的错误信息给调用方
  response->set_success(false);
  response->set_error_code(ErrorCode::PROCESSING_FAILED);
  response->set_error_message(e.what());
}
```

**关键原则：**
- 捕获具体异常类型，而非 `catch(...)`
- 记录包含输入参数、请求 ID、错误类型的完整日志
- 更新诊断状态为 ERROR，附带可解释信息
- 返回结构化的错误响应（错误码 + 错误消息），让调用方能正确处理

1. **违反 §6.2「可测试函数设计」— 优先使用显式返回值**：`ProcessFrame` 通过修改成员变量 `last_result_` 来传递结果，而非通过返回值。测试时无法直接通过 `EXPECT_EQ(result, expected)` 验证，必须额外调用 `GetLastResult()` 间接检查，增加测试复杂度且依赖调用顺序。

2. **隐式副作用降低可推理性**：函数签名为 `void`，调用者无法从签名判断函数会修改哪些状态。`success_count_++` 更是隐藏的副作用，与函数"处理帧"的主要职责无直接关系。

3. **并发安全隐患**：多线程环境下，`last_result_` 和 `success_count_` 的修改没有锁保护，如果其他线程同时调用 `GetLastResult()` 或 `ProcessFrame()`，会产生数据竞争。

```cpp
class FilterProcessor {
 public:

  }
};
```
