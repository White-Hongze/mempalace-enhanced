# ROS2/C++ 编码规范 RAG 召回验证实验

## 实验目标

验证RAG系统能否准确检索和应用《面向 RAG 召回测试的编码规范文档》中的约束条件，覆盖以下维度：

1. **直接术语命中**：通过关键词精准定位规范条目
2. **同义表达理解**：理解问题的多种表达方式
3. **场景推理应用**：在架构设计中正确应用规范
4. **代码缺陷识别**：指出违反规范的代码模式
5. **代码生成合规**：生成符合规范的代码实现

---

## 实验设计

### A. 术语与定义类测试（10题）

#### A1. 临界区最小化原则
**问题**：互斥锁的保护范围中，哪些操作原则上不得放入？
- 预期关键词：I/O操作、日志格式化、网络发送、回调调用、容器遍历

#### A2. 锁顺序一致性
**问题**：为什么多把锁时必须建立全局锁顺序？违反后会怎样？
- 预期关键词：死锁、锁顺序反转、不可预测时延

#### A3. 条件变量用法
**问题**：为什么条件变量必须使用谓词等待？
- 预期关键词：虚假唤醒、等待条件、显式谓词

#### A4. ROS2 回调约束
**问题**：ROS2 订阅回调中禁止做哪些长阻塞操作？
- 预期关键词：长时间sleep、磁盘写入、网络请求、长时间等待锁

#### A5. 动态参数更新
**问题**：参数动态更新时为什么要用快照语义？
- 预期关键词：一致性、半更新配置、工作线程、一次性生成

#### A6. 依赖注入
**问题**：为什么不要在业务类内直接new外部依赖？
- 预期关键词：可mock、可替换、隔离测试、RAG识别边界

#### A7. 错误处理
**问题**：错误日志必须包含哪些信息才能有效定位问题？
- 预期关键词：关键字段、frame_id、request_id、状态机阶段、错误码

#### A8. 背压与队列
**问题**：无上限队列缓存的风险是什么？
- 预期关键词：内存溢出、无控制增长、生产消费不平衡、丢包策略

#### A9. 诊断状态
**问题**：诊断状态与业务状态为什么要分离？
- 预期关键词：明确映射、INIT、WAITING、RUNNING、DEGRADED、ERROR、可解释

#### A10. 纯逻辑与ROS分离
**问题**：为什么要把纯逻辑与ROS接口层分离？
- 预期关键词：单元测试成本、自动生成测试、不依赖Node

### C. 场景理解与架构推理类测试（8题）

#### C1. 节点职责过载
**场景**：一个ROS2节点同时负责传感器订阅、数据融合、轨迹规划、诊断聚合、落盘和Web接口输出。

**问题**：这个设计存在哪些问题？如何重构？

**预期答案应涉及**：
- 规范 3.1.1 中的多职责风险（参数耦合、回调相互影响、测试边界不清、故障隔离难）
- 推荐的解耦方案（ROS接口+业务逻辑下沉、独立类、状态管理分离）

#### C2. 高频路径性能
**场景**：一个100Hz订阅回调中：
- 对消息做合法性检查
- 更新mutex保护的内部缓存
- 调用Publisher发送处理结果
- 记录详细日志

**问题**：这个设计会带来什么问题？

**预期答案应涉及**：
- 规范 3.2.1：发布消息可能阻塞或耗时
- 规范 3.2.3：日志没有节流
- 规范 2.1.1：发布操作不应在锁内
- 性能影响和executor堆积

#### C3. 线程安全设计
**场景**：一个模块有3把互斥锁（config_mutex, state_mutex, cache_mutex），在不同函数中被交叉获取。某处代码先拿state后拿config，另处先拿config后拿state。

**问题**：为什么这很危险？应该怎么修复？

**预期答案应涉及**：
- 规范 2.1.3：锁顺序全局一致性
- 死锁风险
- 全局顺序建立的推荐方式

#### C4. 参数一致性
**场景**：某个模块有多个动态参数（timeout_ms, max_retries, enable_feature）。工作线程在处理时分别读取这三个参数，中间有参数动态更新。

**问题**：可能出现什么问题？

**预期答案应涉及**：
- 规范 4.1.2：快照语义
- "半更新配置"的风险
- Config结构体 + 快照的解决方案

#### C5. 背压管理
**场景**：一个Producer线程向queue投递数据，Consumer线程处理。Producer速度常常大于Consumer。当前队列无上限增长。

**问题**：这会导致什么问题？应该设计哪些机制？

**预期答案应涉及**：
- 规范 7.2：背压与队列管理
- 内存溢出、无控制增长、反压失控
- 队列上限、丢旧/丢新策略、最新值覆盖、丢包统计、诊断告警

#### C6. 接口与可测试性
**场景**：一个算法处理类在内部直接new了Clock、Logger、FileWriter、MetricsClient，所有计算逻辑掺杂ROS2 Node依赖。

**问题**：为什么单元测试很困难？如何改进？

**预期答案应涉及**：
- 规范 6.1.1：依赖注入
- Mockable接口、构造函数注入
- 规范 6.2.1：纯逻辑与ROS分离

#### C7. 诊断状态设计
**场景**：某个模块的诊断只输出一个布尔值 `is_healthy`，当为false时无法判断问题所在。

**问题**：应该怎么改进诊断输出？

**预期答案应涉及**：
- 规范 5.2.1：状态与业务状态分离（INIT, WAITING, RUNNING, DEGRADED, ERROR, STOPPED）
- 规范 5.2.2：诊断可解释性（状态、原因、影响范围、恢复条件）

#### C8. 错误恢复
**场景**：某个Service服务处理请求时异常，代码直接catch后返回空结果。

**问题**：问题在哪？应该怎么做？

**预期答案应涉及**：
- 规范 5.1.1：禁止吞掉异常
- 规范 5.1.2：错误日志要携带可定位信息（frame_id、request_id等）
- 降级策略、上报、传播决策

### D. 代码违规识别类测试（10题）

为每个代码片段提供有缺陷的C++代码，要求AI识别违反的规范编号和问题。

#### D1. 持锁调用外部代码
```cpp
{
  std::lock_guard lock(mutex_);
  publisher_->publish(msg);  // 在锁内发布消息
  external_callback();        // 在锁内调用外部函数
  ProcessData();              // 复杂处理
}
```
**预期识别**：规范 2.1.1（持锁区过大）、2.1.2（在锁内调用外部代码）

#### D2. 异常吞掉
```cpp
try {
  LoadConfiguration();
  ValidateParameters();
  InitializeModule();
} catch (...) {
  // 什么都不做
}
```
**预期识别**：规范 5.1.1（吞掉异常）、5.1.2（没有诊断信息）

#### D3. 锁顺序反转
```cpp
// 函数A
{
  std::lock_guard lock1(mutex_a);
  std::lock_guard lock2(mutex_b);
  // ...
}

// 函数B
{
  std::lock_guard lock2(mutex_b);
  std::lock_guard lock1(mutex_a);  // 顺序反转！
  // ...
}
```
**预期识别**：规范 2.1.3（锁顺序全局一致）

#### D4. 回调中长阻塞
```cpp
void ImageCallback(const Image::SharedPtr msg) {
  std::this_thread::sleep_for(std::chrono::seconds(2));
  
  file_writer_->Write(msg->data);  // 磁盘写入
  
  auto result = heavy_compute(msg);
  
  RCLCPP_INFO(logger_, "Processed frame %d", msg->frame_id);
}
```
**预期识别**：规范 3.2.1（回调中禁止长时间阻塞）、3.2.3（日志没有节流）

#### D5. 无谓词条件变量
```cpp
void WaitForReady() {
  std::unique_lock lock(mutex_);
  cv_.wait(lock);  // 没有谓词，容易虚假唤醒
  // 假设condition满足，但实际可能不满足
}
```
**预期识别**：规范 2.3.1（必须使用谓词等待）

#### D6. 多个原子变量伪同步
```cpp
std::atomic<bool> has_data{false};
std::atomic<int> frame_id{0};
std::atomic<int64_t> timestamp{0};

void UpdateData(Frame f) {
  has_data.store(true);
  frame_id.store(f.id);      // 三个变量独立更新
  timestamp.store(f.ts);
}
```
**预期识别**：规范 2.2.1（多字段一致性不能靠多个原子变量拼装）

#### D7. 零散读参数
```cpp
void ProcessData() {
  auto timeout = config_.get<int>("timeout_ms");
  auto retries = config_.get<int>("max_retries");
  
  // ... 处理逻辑...
  
  auto enabled = config_.get<bool>("enable_feature");
}
```
**预期识别**：规范 4.1.2（参数更新要用快照，不要零散读取）

#### D8. 没有最新值缓存管理
```cpp
std::deque<Message> queue;

void callback(const Message::SharedPtr msg) {
  queue.push_back(*msg);  // 无上限增长，无背压
}

void process() {
  while (!queue.empty()) {
    auto msg = queue.front();
    queue.pop_front();
    Process(msg);
  }
}
```
**预期识别**：规范 3.3.2（没有明确最新值vs全量事件）、规范 7.2（无上限队列、无背压策略）

#### D9. 节点职责过载
```cpp
class VisionProcessorNode : public rclcpp::Node {
  // 订阅图像
  subscription_ = create_subscription<Image>(...);
  // 订阅IMU
  imu_sub_ = create_subscription<Imu>(...);
  // 融合与规划
  planner_ = new Planner();
  // 诊断聚合
  diag_pub_ = create_publisher<DiagnosticArray>(...);
  // 落盘
  file_writer_ = new FileRecorder();
  // Web接口
  web_server_ = new WebServer();
};
```
**预期识别**：规范 3.1.1（单个节点职责过多）

#### D10. 隐式副作用设计
```cpp
class FilterProcessor {
 public:
  void ProcessFrame(const Frame& f) {
    auto filtered = ApplyFilter(f);
    last_result_ = filtered;  // 悄悄修改成员变量
    success_count_++;          // 维护计数器
  }
  
  Frame GetLastResult() { return last_result_; }
};
```
**预期识别**：规范 6.2.2（隐式副作用，不利于测试）

## 评分标准

### A. 术语与定义类（10题，每题1分）
- **满分**：准确列举出规范中的关键词/约束条件
- **部分分**：列举出大部分关键词
- **零分**：无关或完全错误

### B. 同义改写类（10题，每题2分）
- **2分**：准确理解题意，指出规范编号，解释关键约束和原因
- **1分**：能指出相关规范但解释不完整
- **0分**：无关或错误

### C. 场景推理类（8题，每题3分）
- **3分**：准确指出问题、涉及的规范编号(≥2条)、解决方案有实际可行性
- **2分**：能指出问题和1-2条规范，但方案不具体
- **1分**：能指出有问题但解释肤浅
- **0分**：完全偏离或错误

### D. 代码违规识别（10题，每题2分）
- **2分**：准确指出违反的规范编号（≥2条）和具体问题
- **1分**：能指出问题但规范编号不准确
- **0分**：无法识别或错误

### E. 代码生成（5题，每题5分）
- **5分**：生成代码完整有效，遵循所有相关规范（≥3条规范）
- **4分**：代码基本可用，遵循大部分规范（2-3条）
- **3分**：代码有框架但细节不规范（1-2条）
- **2分**：代码有明显缺陷或违反规范
- **1分**：代码框架错误或规范理解严重偏差
- **0分**：无法生成或完全错误

### 总分计算
```
总分 = A(10×1) + B(10×2) + C(8×3) + D(10×2) + E(5×5)
     = 10 + 20 + 24 + 20 + 25
     = 99分

评级标准：
- 90分及以上：优秀（RAG召回效果好，规范理解准确）
- 75-89分：良好（RAG基本工作，有部分理解偏差）
- 60-74分：及格（RAG能工作但易遗漏细节）
- 60分以下：需改进（RAG召回不足或错误率高）
```

## 分析维度

### 1. 召回准确度（Precision）
- 检索出的规范条目是否与问题相关
- 公式：正确检索 / 总检索数

### 2. 召回完整度（Recall）
- 在所有相关规范中，检索到的比例
- 公式：检索到的相关规范 / 所有相关规范

### 3. 应用正确率
- 在实际代码审查和生成中正确应用规范比例

### 4. 规范覆盖率
- 被RAG系统识别和应用的规范条目占比
- 通常期望 > 80%

### 5. 细节化程度
- 是否能指出规范的具体条款（如 2.1.1 vs 笼统的"锁的规范"）
- 评估系统对文档结构的理解

## 扩展建议

### 对话连贯性测试
多轮问答中，AI是否保持对规范的理解一致性：
- Q1: 询问规范A
- Q2: 基于A的理解，延伸提问
- 检查是否出现自相矛盾

### 边界条件测试
- 问题的复杂混合：一个问题涉及多条规范的交互
- 反向问题：什么情况下规范不适用
- 权益冲突：当两条规范产生张力时如何平衡

### 文本变体测试
相同问题的多种表述：
- 正式学术用语
- 口语化工程师语言
- 代码注释形式
- 故障现象描述

验证RAG对表达变体的鲁棒性。
